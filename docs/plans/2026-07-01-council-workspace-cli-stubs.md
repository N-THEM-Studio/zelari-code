## Goal

Make the Zelari Code **council** (multi-agent orchestration) fully autonomous
in CLI standalone. Currently the council produces excellent analysis/ideation
text but its workspace tools (`createPhase`, `addIdea`, `createMilestone`,
etc.) return errors because they expect an AnathemaBrain Electron context.
The agent's reasoning is **volatile** â€” only seen on screen, never persisted.

This plan adds **coding-flavored** workspace stubs that persist to
**`.zelari/`** (project-local, gitignored) so the council becomes a
top-tier coding planning partner.

## Design Philosophy

**Coding-first**, not workspace-generic. We're not building a Notion
clone â€” we're building a **coding agent's thinking space**:

- An ADR (`addIdea`) means *"we decided X because Y, traded off Z"*
- A phase (`createPhase`) means *"milestone toward shipping this feature"*
- A milestone (`createMilestone`) means *"the code version that ships when X works"*
- A doc (`createDocument`) means *"a README/CHANGELOG/spec draft the council produced"*
- A search (`searchDocuments`) means *"what did we decide earlier about X?"*

## AGENTS.MD Auto-Maintenance

The crown jewel of the workspace: a **`AGENTS.MD`** at project root that the
council **auto-curates** as it accumulates project directives and architecture.
This file becomes the canonical "operating manual" for any AI agent (Zelari
Code itself, future agents, or human contributors) that touches the codebase.

### Why AGENTS.MD

The standard is emerging (Claude Code, Aider, Cursor, Continue all read it).
For Zelari Code â€” a top-tier coding agent â€” auto-curating AGENTS.MD turns
council runs into **self-documenting project intelligence**. Every directive
the council agrees on becomes part of the project's permanent operating spec.

### File location & git status

- **Path**: `AGENTS.MD` (project root, **NOT inside `.zelari/`**)
- **Git status**: **tracked** (committed) â€” this is a deliverable, not session state
- **Format**: pure Markdown, no frontmatter (committable, renderable on GitHub)

### Update strategy: marker sections

AGENTS.MD uses **HTML comment markers** to delimit auto-managed blocks. Manual
edits inside markers are preserved across updates; manual sections outside
markers are never touched.

| `decisions` | `.zelari/decisions/NNN-*.md` with `status: accepted` | Every council run that promotes a decision |

### Trigger flow

```
/council <task>
   â†“
council run completes (chairman synthesizes)
   â†“
post-run hook (new file: src/cli/workspace/postCouncilHook.ts)
   â†“
1. Read .zelari/decisions/ â†’ filter accepted ADRs
2. Read .zelari/risks.md    â†’ extract info-level items
3. Read .zelari/plan.md     â†’ current scope context
4. For each auto-managed section in AGENTS.MD:
     a. Parse existing block (between markers)
     b. Generate new block content from current artifacts
     c. If different â†’ update block
     d. If unchanged  â†’ no-op (don't dirty git)
5. Write AGENTS.MD only if at least one block changed
   â†“
chat reply: "AGENTS.MD updated: 2 sections changed (decisions, conventions)"
```

### Idempotency guarantees

- **Content-addressed sections**: each block has a hash of its desired content
- If the existing block matches the hash â†’ no write (no spurious diff)
- If `AGENTS.MD` doesn't exist â†’ create with all 5 sections empty
- If markers are missing (manual edit) â†’ treat the whole file as manual, skip auto-update (fail-safe)

**Project-root `AGENTS.MD`** (tracked/committed) â€” the **deliverable** that
auto-curates from `.zelari/` artifacts (see [AGENTS.MD Auto-M](#agentsmd-auto-maintenance)).

**Why Markdown + YAML frontmatter (no deps):**
- Human-readable, git-diff-friendly
- Renders on GitHub natively
- Custom mini-YAML-parser (~50 LOC) keeps zero new deps (HARD rule)
- Frontmatter carries typed metadata (status, refs, dates)

**Why project-local + auto-gitignored:**
- Council output is **session-local thinking**, not project artifact
- Doesn't pollute git history with ephemeral decisions
- Per-session: `.zelari/` regenerated each `/council` run if missing
- If user wants to keep decisions: they can `cp -r .zelari ~/keep/zelari-feature-x`

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| YAML parser bugs on edge cases | Use only flat key-value (no nested structures in frontmatter); validate with Zod schema |
| Concurrent writes corrupt files | Per-path async mutex (~15 LOC) |
| `.zelari/` pollutes user repos | Auto-create `.zelari/.gitignore` with `*\n!.gitignore` on first write |
| Council tool descriptions drift from CLI semantics | Update `tools.ts` descriptions to match new schema (one-time edit) |
| Workspace grows unbounded | Soft cap: `.zelari/workspace.json` tracks total size; warn if >5MB |
| User wants persistent ADRs (not session-local) | `/workspace export` command â†’ copies `.zelari/` to `docs/decisions/` (committable) |
