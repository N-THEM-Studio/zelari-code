## Architettura Fondamentale (First Principles)

### 1. `@zelari/core` come package pubblico

**Stato attuale:** il codice è *logicamente* separato (commenti "Electron-free" in AgentHarness, directory `src/main/core/` che è solo lessicale) ma non c'è **package boundary** fisico. `src/agents/` (council) e `src/main/core/` (harness + tools) vivono nello stesso package del CLI.

**Cosa serve:**
- Decidere: **monorepo (npm workspaces)** vs **repo separato** (`N-THEM-Studio/zelari-core`)
  - Monorepo: più semplice, sync, ma vincola versioning
  - Repo separato: versioning indipendente, ma +1 layer di release management
  - **Raccomandazione: monorepo** (Turbo o pnpm workspaces) per ora — è un singolo team, versioning indipendente si può sempre fare dopo
- Creare `packages/core/` con:
  - `src/main/core/` → `packages/core/src/harness/`
  - `src/main/core/tools/` → `packages/core/src/tools/`
  - `src/agents/` (councilApi, roles, promoteMember, skills) → `packages/core/src/council/`
  - `src/agents/skills/builtin/` → `packages/core/src/skills/`
  - `src/shared/` (BrainEvent types) → `packages/core/src/events/`
  - `src/types/` (workspace, context) → `packages/core/src/types/`
- Package `zelari-code` diventa thin wrapper: import `@zelari/core`, gestisce solo CLI/Ink UI

### 2. AgentHarness come singolo punto di verità

Già vero. **Validare con audit**:
- Tutti i consumer (CLI, council, future GUI) passano per `AgentHarness`?
- Le dipendenze `electron/` nei commenti sono rimosse o sono residui?
- L'API pubblica è stabile e documentata?

### 3. Thin CLI wrapper

Già parzialmente vero. Verificare:
- `src/cli/app.tsx` (175 LOC post v0.4.2) → solo wiring
- Tutta la logica di business è in `@zelari/core`?
- Niente state machine complessa nella UI (dovrebbe essere in core)?

### Fase 3: Reliability & Truth-Seeking (1-2 settimane)

**Goal:** `/council` affidabile, visibile, verificabile.

- [ ] **Headless mode**
  - `zelari-code --headless --task "refactor foo.ts" --output json`
  - No Ink, no TUI → stdout JSON stream
  - CI-friendly: exit code 0/1, log strutturato
  - Use case: GitHub Action, pre-commit hook, cron
- [ ] **Miglior grounding**
  - Ogni claim del council deve citare almeno 1 file:riga del workspace (o marcare "speculative")
  - Validazione automatica: se Minosse (critic) cita file inesistente → warning

## Rischi & Anti-pattern

| Rischio | Mitigation |
|---------|------------|
| Monorepo setup rompe la CI | Feature branch, dry-run CI su 5 commit, fallback a repo singolo se bloccante |
| Wizard troppo lungo → peggiora onboarding | Max 5 step, skip sempre possibile, default sensati |
| Visible reasoning troppo verbose → overwhelming | Toggle off di default, opt-in per ruolo |
| `@zelari/core` come API pubblica blocca refactor futuri | Versioning semver stretto, internal modules in `core/internal/` non esportati |
| Audit agy trova bug in Fase 0 che sbloccano tutto | Normale — è il punto dell'audit. Allocare 2-3 giorni buffer |
| Council reliability regression post headless mode | E2E test suite obbligatoria, no merge senza 10/10 verde |
