## Issue 2 — `grep_content` non ricorsivo (FUNZIONALE)

### Problema
`src/main/core/tools/builtin/search.ts:6-11`:
```ts
const GrepContentArgsSchema = z.object({
  path: z.string().min(1),  // ← singolo file
  pattern: z.string().min(1),
  ...
});
```

Il parametro si chiama `path` ma semanticamente è **un singolo file**. Non accetta:
- Directory (es. `path: 'src/'` → errore)
- Glob pattern (es. `path: 'src/**/*.ts'`)
- Multi-file (es. `path: 'src/cli/'` → errore)

Questo è un limite grosso per coding agent: per cercare "tutte le occorrenze di X nel progetto" deve fare N chiamate read+grep_content. Pattern Hermes usa `fff_grep` (directory + glob).

### Fix (2-3 ore)
Estendere lo schema e l'execute per supportare:
1. **path** può essere file o directory
2. Nuovo parametro **`include`**: glob pattern per filtrare i file dentro la dir (es. `['*.ts', '*.tsx']`)
3. Riusare logica `walk()` di `listFiles.ts` per attraversare la dir con depth limit
4. Output invariato ma con `file: <relpath>` per ogni match

### Sketch
```ts
const GrepContentArgsSchema = z.object({
  path: z.string().min(1),                              // file O directory
  pattern: z.string().min(1),
  include: z.array(z.string()).default(['*']),          // glob filter
  exclude: z.array(z.string()).default([                 // reuse listFiles defaults
    'node_modules', '.git', 'dist', 'build', '.next', '__pycache__', '.cache',
  ]),
  contextLines: z.number().int().nonnegative().default(2),
  maxMatches: z.number().int().positive().max(1000).default(50),
  maxDepth: z.number().int().positive().max(10).default(8),
});
```

### Verifica
- Test `tests/unit/grep-content-recursive.test.ts`:
  - directory con 5 file .ts, 3 file .md → cerca pattern, ottiene solo .ts
  - directory vuota → result vuoto, no error
  - path singolo file (legacy) → comportamento invariato
  - exclude `node_modules` non viene mai attraversato
- Backward compat: chiamate esistenti con `path: 'src/main.ts'` continuano a funzionare

## Issue 3 — Diff-aware edits (FUNZIONALE)

### Problema
- `edit_file` esiste ed è già string-replace (`oldString/newString/replaceAll`) in `src/main/core/tools/builtin/filesystem.ts:85-134`
- **MA** non mostra diff visivo: l'utente vede solo "occurrencesReplaced: 3" senza sapere cosa è cambiato
- Non c'è fuzzy match: se il modello sbaglia di 1 char → 0 occurrences → silenzioso fail
- Non c'è modalità "line range" (es. "replace lines 10-15 con nuovo contenuto")

Pattern Aider/Cursor: prima mostra diff, poi applica. Pattern Cline/Roo: `apply_diff` con fuzzy.

### Fix (3-4 ore)
Aggiungere 2 nuovi tool in `src/main/core/tools/builtin/diff.ts`:

#### Tool 1: `show_diff`
Mostra unified diff senza modificare file.
```ts
{
  name: 'show_diff',
  description: 'Show unified diff between current file content and proposed content. Read-only.',
  inputSchema: z.object({
    path: z.string(),
    proposedContent: z.string(),
    contextLines: z.number().int().nonnegative().default(3),
  }),
  execute: ... // return { diff: '<unified diff string>', hunks: [...] }
}
```

#### Tool 2: `apply_diff`
Applica un diff in formato unified (parse + apply). Fuzzy tolerance opzionale.
```ts
{
  name: 'apply_diff',
  description: 'Apply a unified diff patch to a file. Returns the new content + diff summary.',
  inputSchema: z.object({
    path: z.string(),
    diff: z.string(),                        // unified diff format
    fuzzyMatch: z.boolean().default(false),  // 1-char tolerance
    dryRun: z.boolean().default(false),      // mostra diff senza applicare
  }),
  execute: ... // parse diff, apply hunks, return { applied: N, skipped: N, finalContent: string }
}
```

#### Bonus: estendere `edit_file` esistente
Aggiungere campo `preview?: boolean` che restituisce il diff senza applicare.

### Verifica
- Test `tests/unit/diff-tools.test.ts`:
  - `show_diff` con contenuto identico → diff vuoto, hunks vuoti
  - `show_diff` con 1 riga cambiata → 1 hunk con -/+
  - `apply_diff` con diff valido → applica, ritorna nuovo contenuto
  - `apply_diff` con diff malformato → errore typed, no partial apply
  - `apply_diff` con `fuzzyMatch: true` → tollera 1 char di differenza

## Note implementative

### Per grep_content recursive
- Riusare `walk()` di `listFiles.ts` estraendola in un helper condiviso (`src/main/core/tools/builtin/_walk.ts`)
- Non aggiungere dipendenze (no `globby`, no `fast-glob`) — regex custom è sufficiente per i pattern semplici
- `exclude` default copia quelli di `list_files` per consistenza

### Per diff tools
- `show_diff`: usare `diff` package nativo di Node? No — Node non ha diff nativo. Usare implementazione custom ~80 LOC (algoritmo Myers semplificato) oppure aggiungere `diff` (1 dep, ~30kb).
- **Decisione**: aggiungere `diff@^5.2.0` come dipendenza. È maturo, MIT, zero deps transitive, ~30kb.
- `apply_diff`: parsing unified diff custom (~50 LOC), validazione hunks, applicazione con fuzzy match opzionale
- Visualizzazione diff in CLI: riusare `CollapsibleToolOutput` (esiste già in `src/cli/components/`)

### Backward compatibility
Tutte le modifiche sono additive o estendono campi opzionali. Nessun tool esistente cambia signature.
